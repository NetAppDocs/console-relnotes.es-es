=== Cloud Volumes ONTAP



==== Posibilidad de gestionar la configuración del proxy del agente de NetApp Console para Cloud Volumes ONTAP

Ahora puedes gestionar la configuración del proxy en el agente de la NetApp Console para Cloud Volumes ONTAP, incluso si pierdes la conexión o el proxy está configurado incorrectamente. Antes, si el agente de la NetApp Console no podía conectarse a Cloud Volumes ONTAP durante 20 minutos, sobrescribía tu configuración manual del proxy con la configuración predeterminada. Esto causaba fallos de comunicación, incluyendo problemas con los mensajes de AutoSupport. Para conservar tu configuración del proxy para los sistemas existentes, ejecuta la siguiente llamada a la API:

`PUT /occm/config`

Incluye el siguiente parámetro en el cuerpo de la solicitud:

[source, json]
----
{
  "proxyMode": "No_Overwrites"
}
----
El modo predeterminado es estándar, lo que significa que el agente de la NetApp Console sobrescribe tu configuración de proxy con los valores predeterminados si no puede conectarse a Cloud Volumes ONTAP durante 20 minutos.

https://docs.netapp.com/us-en/console-automation/cvo/put-occm-config.html["Configura los parámetros modificables de NetApp Console"^]

=== Copia de seguridad y recuperación



==== Mejoras en las cargas de trabajo de ONTAP Volumes

En esta versión, ONTAP Volumes introduce las siguientes capacidades mejoradas:

* *Restaurar en la ubicación de origen*: Ahora puedes restaurar archivos de instantáneas locales e instantáneas replicadas en la ubicación de origen, así como en una ubicación alternativa.
* *Descubrimiento de volúmenes por hora*: el Dashboard ahora descubre y actualiza nuevos volúmenes más rápido.
* *Soporte para dark sites*: Ahora puedes descargar Backup and Recovery for dark sites desde el  https://mysupport.netapp.com/site/products/all/details/netapp-console/downloads-tab/download/61906/4.4.0/downloads["Sitio de soporte de NetApp"]. En este momento, NetApp Backup and Recovery for dark sites solo admite cargas de trabajo de ONTAP Volumes.
* *Mejoras de compatibilidad con Unicode*: la indexación de catálogos ahora maneja mejor los caracteres Unicode.
* *Compatibilidad mejorada con FlexGroup*: Cuando conviertes un volumen ONTAP FlexVol a un volumen FlexGroup, ahora puedes restaurar las copias de seguridad hechas antes de la conversión.
* *Compatibilidad con varios buckets*: ahora puedes proteger los volúmenes dentro de un sistema con hasta 6 buckets por sistema a través de diferentes proveedores de nube.


Para obtener detalles sobre cómo proteger las cargas de trabajo de ONTAP Volume, consulta https://docs.netapp.com/us-en/data-services-backup-recovery/prev-ontap-protect-overview.html["Descripción general de la protección de cargas de trabajo de volúmenes de ONTAP"].



==== Mejoras en las cargas de trabajo de VMware

En esta versión, VMware workloads introduce las siguientes capacidades mejoradas:

* *Restaurar datos de invitados desde el almacén de objetos*: Ahora puedes restaurar archivos y carpetas de invitados desde el almacén de objetos.
* *Restauración rápida para VMFS/NFS*: Ahora puedes activar la opción de restauración rápida al restaurar en una ubicación alternativa. Cuando está activada, solo se recuperan los metadatos de la VM (archivos, LUNs y espacios de nombres) del almacén de objetos, haciendo que los volúmenes estén disponibles más rápido que en una restauración completa.
* *Soporte para Amazon FSx for NetApp ONTAP*: Las cargas de trabajo de VMware ahora son compatibles con Amazon FSx for NetApp ONTAP storage systems que ejecuten la versión 9.13.1 o posterior.


Para obtener detalles sobre cómo proteger las cargas de trabajo de VMware, consulta https://docs.netapp.com/us-en/data-services-backup-recovery/br-use-vmware-protect-overview.html["Descripción general de la protección de cargas de trabajo de VMware"].



=== Clasificación de datos



==== Versión 1.52

.Validación de recursos
Para garantizar la fiabilidad, Data Classification ahora valida la memoria disponible y los recursos de disco durante el escaneo, la generación de informes y las acciones. Cuando el uso de memoria supera el 95% o el espacio en disco cae por debajo de 6 GB, Data Classification pausa automáticamente las operaciones. Las operaciones se reanudan una vez que los recursos son suficientes.

Para monitorear tus recursos del sistema, mira link:https://docs.netapp.com/us-en/data-services-data-classification/task-health-monitor.html["Supervisa el estado de salud de Data Classification"^].

.Límite aumentado para copiar y mover
Las funciones de copiar y mover ahora son compatibles con destinos CIFS. El límite de tamaño por archivo para ambas acciones ha aumentado a 1 GB desde el límite anterior de 50 MB. Los límites de operación para ambas acciones han aumentado a 1 TB o 5 millones de archivos.

Para más información, consulta link:https://docs.netapp.com/us-en/data-services-data-classification/task-copy-move.html["Copiar, mover y borrar datos"^].

.Permisos de configuración de integración de Active Directory
Los cambios en la configuración de integración de Active Directory (AD) en Data Classification ahora están restringidos a los administradores. Los viewers ya no pueden modificar la configuración de AD.

Para obtener más información sobre la función Classification viewer, consulta link:https://docs.netapp.com/us-en/console-setup-admin/reference-iam-predefined-roles.html#data-service-roles["Roles de servicios de datos"^].

.Soporte para RHEL 9.7
Data Classification ha añadido Red Hat Enterprise Linux v9.7 para la instalación manual on-premises, incluidas las implementaciones en dark site.

Los siguientes sistemas operativos requieren usar el motor de contenedores Podman y requieren Data Classification versión 1.30 o superior: Red Hat Enterprise Linux versión 8.8, 8.10, 9.0, 9.1, 9.2, 9.3, 9.4, 9.5, 9.6 y 9.7.

.Soporte de proxy transparente
Data Classification ahora admite configuraciones de proxy transparentes, lo que permite una implementación más fluida en entornos de red con proxy o restringidos.

Para más información sobre la implementación en entornos restringidos, consulta link:https://docs.netapp.com/us-en/data-services-data-classification/task-deploy-compliance-dark-site.html["Instala NetApp Data Classification en un host Linux sin acceso a internet"^].

=== Cloud Volumes ONTAP



==== Compatibilidad de Cloud Volumes ONTAP con la próxima generación de máquinas virtuales de Google Cloud

Con 9.18.1, NetApp realiza la transición de las nuevas implementaciones de Cloud Volumes ONTAP de las máquinas virtuales N2 a la nueva generación de máquinas virtuales de la serie C3 de Google Cloud para una experiencia más rápida y escalable. Ahora puedes aprovechar las máquinas virtuales de la serie C3 al implementar Cloud Volumes ONTAP 9.18.1 y versiones posteriores en Google Cloud. Las máquinas de la serie C3 ofrecen un rendimiento mejorado y límites de capacidad superiores mediante el uso de Google Virtual NIC (gVNIC) y discos Hyperdisk Balanced que aseguran un rendimiento dinámico para cargas de trabajo intensivas.


NOTE: Por ahora, Cloud Volumes ONTAP solo es compatible con la serie C3 en despliegues de un solo nodo.

Si tu sistema Cloud Volumes ONTAP ejecuta la versión 9.18.1 o posterior, los paquetes preconfigurados que usas para implementaciones sencillas de un solo nodo utilizan automáticamente máquinas virtuales C3, y te permiten personalizar los parámetros de IOPS y rendimiento según las necesidades de tu carga de trabajo. De manera similar, al crear agregados, puedes añadir discos Hyperdisk Balanced para lograr un mejor rendimiento y escalabilidad en Google Cloud. Además, puedes seleccionar las variantes LSSD de las máquinas de la serie C3 para compatibilidad predeterminada con Flash Cache.

No puedes cambiar el tipo de disco para las VMs C3 cuando añades volúmenes a agregados porque C3 solo admite discos Hyperdisk Balanced. De manera similar, cuando replicas un sistema con tipos de VM N2 a una VM C3, el tipo de disco se establece por defecto en Hyperdisk Balanced.

https://docs.netapp.com/us-en/cloud-volumes-ontap-relnotes/reference-configs-gcp.html["Configuraciones compatibles para Cloud Volumes ONTAP en Google Cloud"^]

https://docs.cloud.google.com/compute/docs/general-purpose-machines#c3_series["Google documentos: serie de máquinas C3"^]



==== Seguridad VNet para Cloud Volumes ONTAP en Azure

Cloud Volumes ONTAP 9.18.1 y las implementaciones posteriores en zonas de disponibilidad única y múltiple de Azure admiten el cifrado de Azure Virtual Network (VNet) como parte de su estrategia de seguridad por capas para proteger los datos en tránsito. Cloud Volumes ONTAP aprovecha el protocolo nativo Datagram Transport Layer Security (DTLS) de Azure para proteger la comunicación entre los nodos ONTAP, las interfaces de gestión y otros servicios de Azure, evitando la interceptación y el acceso no autorizado. Este cifrado a nivel de red complementa las protecciones integradas de almacenamiento y datos en reposo de ONTAP para ofrecer seguridad de extremo a extremo a tus datos.

https://review.docs.netapp.com/us-en/storage-management-cloud-volumes-ontap_gcp-sn-c3-support/reference-networking-azure.html#networking-for-azure-vnet-encryption["Redes para el cifrado de Azure VNet"^]

Esta versión incluye lo siguiente:

NetApp Console admite la asignación de roles a grupos federados (por ejemplo, grupos de Active Directory), lo que permite a los administradores automatizar la incorporación y salida de usuarios en función de la membresía del grupo en el proveedor de identidad (IdP) de la organización. Esta función reduce la sobrecarga administrativa y garantiza un acceso seguro y consistente al actualizar instantáneamente el acceso a la consola a medida que cambian las membresías del grupo.

.Modo de solo lectura
Puede configurar una organización de la NetApp Console en modo de solo lectura. En el modo de solo lectura, los usuarios pueden ver recursos y configuraciones, pero no pueden realizar cambios. === Recuperación ante desastres



==== Compatibilidad con múltiples agentes de consola en entornos locales

Si utiliza recuperación ante desastres en sus instalaciones, ahora puede implementar un agente de consola para cada instancia de vCenter, lo que mejora la resiliencia.

Por ejemplo, si tiene dos sitios (sitios A y B), el sitio A puede tener el agente de consola A conectado a vCenter 1, la implementación de ONTAP 1 y la implementación de ONTAP 2. El sitio B puede tener el agente de consola B conectado a vCenter 2 y a las implementaciones de ONTAP 3 y 4.

Para obtener información sobre el agente de consola en recuperación ante desastres, consulte link:https://docs.netapp.com/us-en/data-services-disaster-recovery/get-started/dr-prerequisites.html#netapp-console-prerequisites["Crear el agente de consola"^].

Debe proporcionar detalles de mapeo para las máquinas virtuales agregadas antes de que se complete la conmutación por error.
