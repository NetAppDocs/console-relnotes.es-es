=== Cloud Volumes ONTAP



==== Presentamos la incorporación simplificada de Cloud Volumes ONTAP desde AWS Marketplace

NetApp introduce un proceso de puesta en marcha simplificado para Cloud Volumes ONTAP desde AWS Marketplace, reduciendo los pasos manuales y la fricción en la puesta en marcha.

Anteriormente, para implementar Cloud Volumes ONTAP desde AWS Marketplace era necesario pasar por varias pantallas y configurar manualmente los roles, las políticas y las credenciales. Esto hacía que la puesta en marcha fuera lenta y propensa a errores.

El nuevo proceso de incorporación simplifica tu transición de una suscripción en el marketplace a un entorno de Cloud Volumes ONTAP totalmente operativo con las siguientes características:

. Suscripción a la ficha específica de Cloud Volumes ONTAP en AWS Marketplace: te suscribes a la ficha independiente de Cloud Volumes ONTAP en AWS Marketplace, compras una suscripción y configuras tu cuenta de AWS (si aún no tienes una).
. Ruta guiada en la NetApp Console: Configuras tu cuenta en la NetApp Console y la Console te incorpora automáticamente al recorrido de implementación, mostrando la ruta guiada en el lado derecho. Si necesitas salir de la Console, puedes reanudar el recorrido guiado más adelante.
. Asociar tu suscripción: Puedes asociar automáticamente tu suscripción con la Consola, evitando los pasos manuales de vincularla tú mismo.
. Implementación del agente de la consola con configuración de credenciales de AWS: puedes crear automáticamente los roles de AWS necesarios. El proceso implementa una pila de AWS CloudFormation en tu cuenta de AWS con los roles y políticas necesarios para que los revises, luego los asocia con la NetApp Console, eliminando la configuración manual de IAM. Luego continúas con la configuración del agente, la red y los ajustes del grupo de seguridad para comenzar la implementación del agente.
. Implementa un nuevo sistema Cloud Volumes ONTAP en AWS: navega directamente al asistente  `Quick create` para una implementación más rápida y sencilla de Cloud Volumes ONTAP.

==== Análisis de configuración well-architected para Cloud Volumes ONTAP en Azure

La NetApp Console ya admite el análisis de configuración well-architected para los sistemas Cloud Volumes ONTAP que se ejecutan en Microsoft Azure. La NetApp Console analiza tus implementaciones diariamente y proporciona un estado well-architected, información detallada y recomendaciones para ayudarte a mantener sistemas que se ajusten a las prácticas recomendadas de NetApp y al marco well-architected.

El panel de control bien estructurado muestra el número total de sistemas analizados y clasifica sus recursos por configuración y nivel de gravedad. Si eres administrador del sistema, puedes ver recomendaciones sobre cualquier problema de configuración dentro de tu organización directamente desde el panel de control de la Console y descargar un informe CSV con todas las recomendaciones. Las configuraciones se clasifican según:

* *Fiabilidad*: Garantiza que las cargas de trabajo funcionen de forma correcta y constante, incluso en caso de interrupciones.
* *Seguridad*: Protege los datos, los sistemas y los activos mediante evaluaciones de riesgos y mitigación.
* *Excelencia operativa*: Optimiza la arquitectura para maximizar el valor empresarial.
* *Optimización de costes*: Reduce y optimiza los costes para generar un mayor valor empresarial.

=== Cloud Volumes ONTAP



==== Cloud Volumes ONTAP 9.19.1 RC

Ahora puedes utilizar la NetApp Console para implementar y gestionar la versión Release Candidate (RC) de Cloud Volumes ONTAP 9.19.1 en Google Cloud. Esta versión no está disponible para implementación ni actualización en Amazon Web Services (AWS) ni en Microsoft Azure (Azure).

link:https://docs.netapp.com/us-en/cloud-volumes-ontap-relnotes/["Más información sobre esta versión de Cloud Volumes ONTAP"^].



==== Compatibilidad con las máquinas virtuales de Google Cloud de última generación en Cloud Volumes ONTAP HA

Con la versión 9.19.1, NetApp traslada las implementaciones de alta disponibilidad (HA) de Cloud Volumes ONTAP de las máquinas virtuales N2 a las máquinas virtuales de la serie C3 de Google Cloud de última generación para ofrecer una experiencia más rápida y escalable.


NOTE: Esta compatibilidad solo se aplica a las implementaciones de alta disponibilidad (HA) en una única zona de disponibilidad (AZ).

Ahora puedes aprovechar las máquinas virtuales de la serie C3 al implementar Cloud Volumes ONTAP 9.19.1 y versiones posteriores en Google Cloud. Estas máquinas ofrecen un rendimiento mejorado y límites de capacidad más altos. Utilizan Google Virtual NIC (gVNIC) y discos Hyperdisk Balanced para un rendimiento dinámico en cargas de trabajo intensivas.

Si tu sistema Cloud Volumes ONTAP ejecuta la versión 9.19.1 o posterior, los paquetes preconfigurados que usas para implementaciones sencillas en AZ únicas utilizan automáticamente máquinas virtuales C3, y te permiten personalizar los parámetros de IOPS y rendimiento según las necesidades de tu carga de trabajo. De manera similar, al crear agregados, puedes añadir discos Zonal Hyperdisk Balanced para lograr un mejor rendimiento y escalabilidad en Google Cloud. También puedes seleccionar las variantes LSSD de las máquinas de la serie C3 para compatibilidad predeterminada con Flash Cache.

Ten en cuenta estos puntos:

* En tus implementaciones de Cloud Volumes ONTAP no puedes cambiar máquinas virtuales C3 por máquinas virtuales de la serie N2, ni viceversa. Sin embargo, puedes cambiar el tipo de instancia entre las diferentes variantes de la familia C3.
+

NOTE: El cambio de tipo de instancia dentro de la familia C3 puede tardar más según la cantidad de discos de datos conectados. Planifica tu configuración con antelación para evitar impactos en el rendimiento.

* Al cambiar el tipo de instancia, si seleccionas c3-standard-44 o c3-standard-88 y sus variantes con SSD local (LSSD) como máquina virtual de destino, se aplica automáticamente el nivel de red de salida del servicio de red de Google _Tier-1_. Si vuelves a cambiar a cualquier otra variante de C3, el nivel de red de salida se establece en _Standard_.
+

NOTE: Cambiar el nivel de red de salida a _Tier-1_ conlleva costes adicionales. Planifica la implementación y la actualización en consecuencia.

* Si seleccionas las variantes LSSD de C3, Flash Cache se activa de forma predeterminada. No puedes cambiar entre las variantes LSSD de C3 en tu implementación de Cloud Volumes ONTAP, incluso si la variante de destino es compatible con Flash Cache.
* En tus máquinas virtuales C3, puedes añadir volúmenes tanto a agregados nuevos como a los ya existentes.
* Al replicar un sistema Cloud Volumes ONTAP con un tipo de máquina virtual N2 a un tipo de máquina virtual C3, el tipo de disco se establece por defecto en Hyperdisk Balanced.
* Puedes modificar los ajustes de IOPS y ancho de banda al añadir agregados a través de la NetApp Console o mediante la  `PUT /gcp/ha/aggregates/{workingEnvironmentId}/{aggregateName}` API.

==== Presentación de los roles de acceso para Cloud Volumes ONTAP

NetApp introduce roles de gestión de identidades y accesos (IAM) para Cloud Volumes ONTAP, lo que permite el control de acceso basado en roles (RBAC) para mejorar la seguridad y facilitar el cumplimiento normativo. Los usuarios pueden seguir iniciando sesión en la NetApp Console utilizando sus credenciales actuales para las operaciones de Cloud Volumes ONTAP. Sin embargo, ahora la autenticación se enruta a través de la federación de identidades, lo que permite al administrador de la Console tener un control más detallado sobre estas funciones. Cada usuario debe tener uno de los siguientes roles para realizar cualquier operación en Cloud Volumes ONTAP. Sin un rol, los usuarios no pueden ver ni acceder a las instancias de Cloud Volumes ONTAP.

* *Cloud Volumes ONTAP viewer*: Otorga acceso de solo lectura a las pantallas y pestañas de Cloud Volumes ONTAP. Ciertas pantallas y botones de acción están deshabilitados en este rol.
* *Cloud Volumes ONTAP operator*: Permite realizar tareas de gestión cotidianas, como añadir volúmenes, gestionar el almacenamiento y configurar la replicación.
* *Administrador de Cloud Volumes ONTAP*: Ofrece acceso completo a todas las operaciones de Cloud Volumes ONTAP.


El administrador de la consola asigna roles a los usuarios de Cloud Volumes ONTAP desde *Administración > Identidad*. Los permisos de los roles se basan en la organización a la que pertenece el usuario y se aplican a todos los sistemas de Cloud Volumes ONTAP de dicha organización.

Cloud Volumes ONTAP aplica el modelo RBAC mediante una caché local de los permisos de cada usuario, que se actualiza automáticamente una vez cada hora. Cuando se asigna un nuevo rol a un usuario o se modifica su rol actual, el cambio puede tardar hasta una hora en surtir efecto, después de lo cual se habilitan todos los flujos de trabajo permitidos por los roles.


NOTE: Cuando se implementa un agente de Console por primera vez, o cuando se actualiza un agente de Console existente a la versión 4.9.0 o posterior, un proceso de configuración inicial único asigna automáticamente roles de Cloud Volumes ONTAP a todos los usuarios existentes de la organización. Este proceso puede tardar hasta 30 minutos en completarse. Y cuando termina, Console asigna los roles adecuados a los usuarios existentes.

Las funciones de Cloud Volumes ONTAP aplican a los entornos de Amazon Web Services (AWS), Microsoft Azure (Azure) y Google Cloud.

==== Actualizaciones de Kubernetes

Disaster Recovery ha introducido los siguientes cambios para las cargas de trabajo basadas en Kubernetes:



===== Gestiona Trident Protect desde el menú «Sitios»

Disaster Recovery ahora permite gestionar Trident Protect directamente desde el menú Sites de la interfaz de usuario de Disaster Recovery. Puedes actualizar Trident Protect o eliminarlo de Disaster Recovery, lo que agiliza la gestión de tus recursos de Kubernetes.

Para más información, consulta link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/manage.html["Gestionar sitios"^].



==== Actualizaciones de VMware

Recuperación ante desastres ha introducido los siguientes cambios para las cargas de trabajo basadas en VMware:



===== Conmutación por error en almacenes de datos vacíos

Los planes de replicación que incluyen almacenes de datos vacíos ahora pueden someterse a una conmutación por error con éxito sin excluirlos del plan ni forzar la conmutación por error. Durante la conmutación por error, Disaster Recovery monta automáticamente el almacén de datos vacío en el sitio de recuperación ante desastres, permitiendo añadir máquinas virtuales y protegerlas para las operaciones de conmutación de vuelta.

Para más información, consulta link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/failover.html["Conmutación por error a un sitio remoto"^].



===== vCenter mapeo automático de red

Ahora puedes configurar las relaciones de asignación de red a nivel de vCenter (es decir, fuera del cuadro de diálogo del plan de replicación). Una vez configurado, Disaster Recovery aplica automáticamente estas asignaciones de red cuando añades o editas un plan de replicación, lo que agiliza la gestión de los planes de replicación.

La recuperación ante desastres sigue admitiendo modificaciones específicas por plan. Las asignaciones de red existentes definidas en un plan de replicación pueden seguir editándose por red; estas asignaciones tienen prioridad sobre las definiciones a nivel de vCenter.



==== Restricción del rol de visor de Disaster Recovery

Se ha eliminado la posibilidad de ver la pestaña Upgrade System del rol Disaster Recovery viewer. Este cambio adapta el rol a la experiencia más adecuada para los usuarios con acceso de solo lectura. link:https://docs.netapp.com/us-en/console-setup-admin/reference-iam-disaster-rec-roles.html["Conoce los roles y permisos de usuario en NetApp Disaster Recovery"^].



==== Mejoras del rendimiento del sistema

Recuperación ante desastres ha introducido cambios para reducir el consumo de recursos del sistema y mejorar el rendimiento en todas las operaciones.

=== Asesor digital



==== Revisión del estado de GraphQL: migración y mejoras en la precisión de los datos

El módulo «Health Check» se ha mejorado con una mayor precisión de los datos y recomendaciones de versiones alineadas con los estándares de vigencia. Como parte de esta versión, se ha migrado de la Public API a GraphQL, lo que ofrece un mejor rendimiento, consultas flexibles y soporte a largo plazo.



===== Software recomendado

Las mejoras recomendadas de software en el módulo Health Check ahora proporcionan datos más precisos y coherentes.

* Los recuentos de estantes y unidades que aparecen en la sección de firmware muestran los valores exactos por sistema y ya no incluyen los recuentos de HA Pair.
* Las recomendaciones sobre la versión mínima y la más reciente ahora se calculan a nivel de clúster en función de la versión efectiva del clúster, en línea con los estándares de actualización; antes, estas recomendaciones se calculaban a nivel de nodo.

Para el vencimiento de la suscripción, ahora las alertas se activan en cinco umbrales: 180, 90, 60, 30 y 7 días antes del vencimiento, con la gravedad aumentando de informativa a crítica a medida que se acerca la fecha.

Para el uso de la capacidad, el umbral de alerta de picos ahora se basa en el porcentaje de picos comprometido de tu suscripción en lugar de un 20% fijo. Por ejemplo, si tu pico comprometido es del 40%, la alerta crítica se activa al 140% del uso de la capacidad comprometida.

Para obtener más información, consulta https://docs.netapp.com/us-en/keystone-staas/integrations/monitoring-alert-monitors.html["Ver y crear monitores de alertas para suscripciones de Keystone"].

==== Mayor flexibilidad para las estrategias de protección

Ransomware Resilience ofrece ahora más flexibilidad a la hora de elegir estrategias de detección de ransomware. Anteriormente, todas las estrategias de protección contra ransomware tenían que incluir la detección de cifrado. ==== Soporte para nuevas instancias EC2 para Cloud Volumes ONTAP en AWS

No recomendamos los tipos de instancia m6id.xlarge, m6id.2xlarge, m6idn.xlarge y m6idn.2xlarge de menor tamaño para cargas de trabajo de alto rendimiento o con uso intensivo de memoria. Para estas cargas de trabajo, usa un tipo de instancia con mayor número de núcleos y más memoria.

=== Cloud Volumes ONTAP



==== Facturación flexible para Cloud Volumes ONTAP y servicios de datos en AWS

NetApp ofrece opciones de facturación mejoradas para tus suscripciones a Cloud Volumes ONTAP en AWS, dándote más control sobre cómo pagas la capacidad de Cloud Volumes ONTAP y los servicios de datos de NetApp, como NetApp Backup and Recovery. Antes, todo el uso de Cloud Volumes ONTAP y de los servicios de datos se facturaba junto a través del listado NetApp Intelligent Services (NIS) en el marketplace de AWS.

Debido a los cambios en AWS Enterprise Discount Program (EDP), NetApp ha introducido una suscripción independiente a Cloud Volumes ONTAP en el marketplace de AWS que te permite separar la facturación de la capacidad de Cloud Volumes ONTAP de los servicios de datos. Esta separación te da más flexibilidad para gestionar tus créditos y costos de EDP. NetApp recomienda usar la suscripción independiente para una mejor gestión de los costos de tu uso de Cloud Volumes ONTAP, mientras sigues usando la suscripción NIS para la facturación de los servicios de datos después del 31 de julio de 2026, cuando entren en vigor los requisitos de EDP.

La suscripción independiente a Cloud Volumes ONTAP en AWS solo admite paquetes de licencia Essentials o Freemium basados en capacidad válidos. No se admite el paquete Professional. Puedes seleccionar el método de facturación que mejor se adapte a tus necesidades:

* *Suscripción independiente a Cloud Volumes ONTAP (recomendada)*: Paga el uso de Cloud Volumes ONTAP a través de la suscripción independiente en el marketplace de AWS, mientras sigues facturando los servicios de datos a través de la suscripción NIS. Esta es la opción recomendada para la mayoría de los clientes que tienen tanto suscripciones independientes como NIS asociadas a su cuenta de AWS en la NetApp Console.
* *Suscripción NIS combinada*: Paga tanto Cloud Volumes ONTAP como los servicios de datos juntos a través del listado NIS existente. El enfoque combinado será válido para los clientes existentes que solo tengan suscripciones NIS asociadas a su cuenta de AWS en la NetApp Console. Sin embargo, NetApp recomienda que configures tu preferencia de facturación para tu uso de Cloud Volumes ONTAP y cambies a la suscripción independiente para una mejor gestión de costos.


Dependiendo de tu tipo de suscripción, obtienes estos escenarios:

.Sólo suscripción independiente (nuevos clientes)
Si solo tienes una suscripción independiente de AWS asociada a tu cuenta de AWS en la NetApp Console y no tienes suscripciones de NIS, solo puedes seleccionar el paquete de licencias Essentials durante la implementación. La opción de servicios de datos *Backup and Recovery* no está disponible para ti.

.Tanto las suscripciones independientes como las de NIS (clientes actuales)
Si tienes suscripciones NIS y autónomas asociadas a tu cuenta de AWS, y no has establecido una preferencia de facturación a través de la pestaña *Preferencias de facturación* en la NetApp Console, el uso de Cloud Volumes ONTAP sigue facturándose a través de NIS de forma predeterminada. Esta también es la opción predeterminada para cualquier instancia de Cloud Volumes ONTAP que ejecute el paquete Professional, porque el listado autónomo no admite Professional. Puedes establecer una preferencia de facturación y elegir cómo facturar cada servicio:

* Establece la suscripción independiente como la cuenta de facturación preferida para Cloud Volumes ONTAP (con el paquete Essentials). Si solo tienes el paquete Professional, puedes convertirlo en un paquete Essentials y cobrar tu uso de Cloud Volumes ONTAP configurando la preferencia de facturación. Te recomendamos este enfoque porque te ayuda a gestionar tus créditos EDP de Cloud Volumes ONTAP de manera más eficaz.
* Utiliza la suscripción NIS para facturar los servicios de datos o el uso de Cloud Volumes ONTAP a través del paquete Professional. Durante el despliegue, activa el servicio de datos *Backup and Recovery* y selecciona el paquete de licencia Professional para facturar los servicios de datos a través de la suscripción NIS.


.Sólo suscripción a NIS (clientes actuales)
Esto es aplicable si eres un cliente existente con suscripciones NIS asociadas a la cuenta de AWS. Puedes seguir facturando tanto Cloud Volumes ONTAP como los servicios de datos juntos, tal como lo haces ahora. Sin embargo, NetApp recomienda cambiar al enfoque de suscripción independiente para una mejor gestión de los costes de tu uso de Cloud Volumes ONTAP:

* Suscríbete a un nuevo listado independiente en el portal de AWS.
* Vincula la suscripción independiente a tu cuenta de AWS.
* Establece tu preferencia de facturación en la suscripción independiente para el uso de Cloud Volumes ONTAP con el paquete Essentials.
* (Si procede) Convierte tu paquete Professional en un paquete Essentials y establece la preferencia de facturación.
* Continúa facturando servicios de datos a través de tu suscripción NIS.



NOTE: No se duplica la medición cuando usas varios tipos de suscripción o migras entre listados.

==== Función IAM de administrador de conformidad

Clasificación de datos ahora admite un rol de administrador de cumplimiento para gestionar los permisos en Clasificación de datos. El rol de administrador de cumplimiento no tiene acceso a recursos, incluidos los agentes de Console y los espacios de trabajo. Los usuarios con el rol de administrador de cumplimiento deben estar asociados con al menos otro rol.

Al instalar Data Classification, debes tener el rol de administrador de la plataforma de Console o el rol de administrador de Compliance.

Si eres un usuario existente, el acceso debería continuar normalmente. Por ejemplo, un rol de plataforma que tiene un rol de administrador de carpeta o proyecto tiene acceso existente.

[[ransomware-resilience]]
=== Resiliencia frente al ransomware



==== Mejoras en la restauración limpia

La restauración limpia ahora admite la recuperación en una ubicación alternativa, lo que te permite restaurar volúmenes en diferentes sistemas, máquinas virtuales de almacenamiento y agregados gestionados por el mismo agente de Console. Además, la restauración limpia ha eliminado la limitación de siete días en la búsqueda de instantáneas viables para la recuperación. La restauración limpia ahora continúa buscando en las instantáneas disponibles un punto de recuperación viable independientemente de la distancia desde el evento de ransomware.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore-recover.html["Recupera cargas de trabajo con una restauración limpia"^].



==== Google SecOps soporte SIEM

Ransomware Resilience ahora es compatible con la gestión de información y eventos de seguridad (SIEM) con Google SecOps, lo que te permite enviar datos automáticamente a Google SecOps para agilizar el análisis y la detección de amenazas.

Para obtener información sobre la configuración, consulta link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-siem.html["Conecta NetApp Ransomware Resilience a un SIEM"^].



==== El soporte de Azure NetApp Files ya está disponible de forma general

Ransomware Resilience ahora es compatible con los sistemas Azure NetApp Files, lo que te permite detectar y responder eficazmente a las amenazas de ransomware en Azure NetApp Files. Cuando descubres cargas de trabajo, Ransomware Resilience ahora presenta Azure NetApp Files y las muestra en el panel de protección. El soporte de Ransomware Resilience para Azure NetApp Files incluye estrategias de detección y protección solo con instantáneas. El soporte para Azure NetApp Files ya está disponible de forma general (GA).

Para más información, consulta link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/concept-ransomware-resilience.html["Conoce la resiliencia ante ransomware"^].



==== Soporte SOAR para bloquear usuarios

Ransomware Resilience ahora soporta el bloqueo de usuarios individuales con Google SecOps y Splunk SOAR playbooks. Con esta adición, puedes automatizar el bloqueo de usuarios cuando Ransomware Resilience detecta actividad sospechosa de un usuario, como una violación de datos o la eliminación masiva de datos.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/reference-soar.html["Integra un playbook SOAR"^].
